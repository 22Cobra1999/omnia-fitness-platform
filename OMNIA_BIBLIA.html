💡 Idea de Negocio: OMNIA

🎯 MISIÓN: POTENCIAR EL FITNESS DIGITAL

OMNIA es una plataforma premium diseñada para conectar a los mejores Coaches de fitness con Clientes que buscan resultados reales.

  • Para Coaches: Herramientas profesionales de gestión, monetización automatizada, y branding personal. Eliminamos el caos administrativo para que puedan enfocarse en entrenar.
  • Para Clientes: Una experiencia de entrenamiento inmersiva, seguimiento de progreso en tiempo real y acceso a un marketplace curado de servicios de salud.
  • Modelo de Negocio: SaaS + Marketplace. Suscripciones para Coaches (Free, Básico, Black, Premium) y comisiones por transacciones seguras vía Mercado Pago.

🌀 Flujo Completo de Meets: Creación, Modificación y Cancelación

📌 ESCENARIO 1: Crear Nueva Meet
Client: Selecciona coach → Elige fecha/hora → Click "Confirmar Reserva"
⬇️
INSERT calendar_events { title, start_time, end_time, coach_id, client_id, status: 'scheduled' }
⬇️
INSERT calendar_event_participants { event_id, client_id, rsvp_status: 'pending', invited_by_user_id: client_id }
⬇️
UI: Modal de éxito → Calendario actualizado
📌 ESCENARIO 2: Modificar Meet Pendiente (Cliente Creador)
Client: Click en meet → "Modificar horario" → Selecciona nuevo slot → "Confirmar"
⬇️ VERIFICACIÓN
rescheduleContext detectado → Fetch participant data
⬇️ CONDICIÓN
IF (rsvp_status === 'pending' && invited_by_user_id === client_id)
⬇️ ACCIÓN
UPDATE calendar_events SET start_time, end_time WHERE id = event_id
⬇️
✅ Meet actualizada (NO duplicada) → Modal de éxito
📌 ESCENARIO 3: Modificar Meet Confirmada
Client: Click en meet confirmada → "Reprogramar" → Selecciona nuevo slot
⬇️ VERIFICACIÓN
rescheduleContext detectado → Fetch participant data
⬇️ CONDICIÓN
IF (rsvp_status === 'confirmed' || 'accepted')
⬇️ ACCIÓN
UI: Modal de preview de reschedule → Requiere confirmación del coach
📌 ESCENARIO 4: Cancelar Meet
Client: Click en meet → "Cancelar solicitud" / "Cancelar mi asistencia"
⬇️
UI: Modal de confirmación personalizado (glassmorphism)
⬇️ SI CONFIRMA
UPDATE calendar_events SET status = 'cancelled' WHERE id = event_id
⬇️
UPDATE calendar_event_participants SET rsvp_status = 'cancelled'
⬇️
Toast: "Asistencia cancelada" → Calendario actualizado
🔧 COMPONENTES CLAVE
CalendarView.tsx: handleTimelineClick(), confirmación en modal y day_split
└──
MeetDetailModal.tsx: handleSuggestNewTime(), handleCancel(), confirmCancel()
└──
rescheduleContext: { eventId, originalTitle, originalDescription }

🐛 Bug Crítico Resuelto: Duplicación de meets al modificar horario. El sistema ahora verifica el estado del meet (pending vs confirmed) y el rol del usuario (creador vs invitado) para decidir si actualizar el evento existente o crear una solicitud de reschedule.

🎨 UX Premium: Modal de confirmación personalizado con glassmorphism reemplaza el window.confirm nativo del navegador.

⚡ Optimización de Reprogramación: Para eventos creados por el usuario que aún están pendientes, se utiliza una Actualización Directa (Overwrite) en lugar de crear una nueva solicitud, evitando la fragmentación de la agenda y duplicados innecesarios.

## 9. 📅 Calendario del Coach (Coach) [ACTUALIZADO]

🌀 Workflow: Ciclo de Vida de eMeets (Interno)
📌 FARE 1: CREACIÓN Y DISPONIBILIDAD
Coach: Abre Calendario → Click en Día (Futuro)
⬇️ INTERACCIÓN
UI: Selector de Fecha (Nuevo Modal 2 Columnas: Años | Meses)
Coach: Click en Slot Horario → Selecciona Cliente
⬇️ LÓGICA DE NEGOCIO
useCoachCalendarLogic.ts: handleCreateMeet()
INSERT calendar_events { title, status: 'scheduled', ... }
INSERT participants { rsvp_status: 'pending' (Client), 'confirmed' (Coach) }
📌 FASE 2: ESTADOS VISUALES (SEMÁFORO)
CONFIRMADA
Borde Naranja Intenso (Coach + Client Confirmed)
INVITACIÓN ENVIADA
Borde Beige/Suave (Client RSVP: Pending)
Muestra conteo de participantes (ej. "0/1 Confirmados")
CAMBIO SOLICITADO
Borde Rojo (Reschedule Request Pending)
Muestra "Propuesta: [Nueva Fecha]" en lugar de texto estático
📌 FASE 3: GESTIÓN Y REPROGRAMACIÓN
Coach: Click en Meet → Modal de Detalle
⬇️ ACCIONES
useCoachEvents.ts: Detecta `rsvp_status` y `reschedule_requests`
Opciones: "Cancelar", "Reprogramar" (Si confirmado), "Reenviar Invitación" (Si pendiente)
🛡️ REGLA DE INTEGRIDAD: ELIMINACIÓN VS CANCELACIÓN

Para mantener un calendario limpio y evitar "fantasmas", el sistema aplica la siguiente lógica al cancelar:

  • ELIMINACIÓN (Hard Delete): Si una Meet (no taller) es cancelada por su creador y ningún invitado ha confirmado todavía, el evento se borra permanentemente del calendario y de la Base de Datos.
  • CANCELACIÓN (Soft Delete): Si ya existen invitados confirmados, el evento cambia su estado a cancelled, pero permanece en la DB (y desaparece del front tras filtrado) para mantener el historial de cancelaciones.

⚠️ Arquitectura Modular: La lógica está estrictamente separada:
useCoachEvents.ts: Fetching y suscripción a cambios en DB (Real-time).
useCoachCalendarLogic.ts: Manejo de interacción, validación de fechas y creación.
coach-calendar-screen.tsx: Únicamente renderizado UI (Dumb Components).

⚙️ Motor Adaptativo OMNIA Om_v2

🧠 ARQUITECTURA: REGLAS ID-CENTRIC

El Motor Adaptativo Om_v2 personaliza cada ejercicio de un programa en el momento de compra. En lugar de lógica hardcodeada, extrae multiplicadores de la tabla adaptive_rules_catalog y los cruza con el perfil del cliente.

TABLAS CLAVE
adaptive_rules_catalog — Catálogo de reglas con multiplicadores kg/series/reps
activities.adaptive_rule_ids integer[] — IDs de reglas que el Coach activó para el programa
clients.profile_rule_ids integer[] — IDs de reglas que describen el perfil del cliente
clients.injury_rule_ids integer[] — IDs de lesiones del cliente
FLUJO DE COMPRA ADAPTATIVA
1. Coach configura Matriz (Baja/Media/Alta) → se guardan IDs reales en activities.adaptive_rule_ids
2. Cliente compra el programa → se llama initialize-progress/route.ts
3. El API cruza adaptive_rule_ids ∩ (profile_rule_ids ∪ injury_rule_ids) del cliente
4. Se acumulan multiplicadores k×s×r via producto: exp(SUM(ln(factor)))
5. Se aplica sobre detalle_series de cada ejercicio → se persiste en progreso_cliente

🔍 Query de Auditoría: Impacto de Reglas por Cliente/Programa

AUDIT QUERY — Cruce Programa vs Cliente
-- ¿Qué reglas del Programa impactan al Cliente?
WITH simulacion AS (
    SELECT r.name, r.kg, r.series, r.repetiiiciones as reps,
           (r.id = ANY(c.profile_rule_ids) OR r.id = ANY(c.injury_rule_ids)) as match
    FROM public.activities a
    JOIN public.adaptive_rules_catalog r ON r.id = ANY(a.adaptive_rule_ids)
    CROSS JOIN public.clients c
    WHERE a.id = [ACTIVITY_ID] AND c.id = [CLIENT_UUID]
)
SELECT
    regla as "Regla en el Programa",
    CASE WHEN match THEN '✅ IMPACTA' ELSE '❌ No aplica' END as "Estado",
    kg as "K", series as "S", reps as "R"
FROM simulacion ORDER BY match DESC;

🚀 Query de Simulación de Compra: Progresión Adaptada por Ejercicio

SIMULATION QUERY — Progresión Base vs Adaptada
WITH multiplicadores AS (
    SELECT
        exp(sum(ln(r.kg)))::numeric            as k_total,
        exp(sum(ln(r.series)))::numeric         as s_total,
        exp(sum(ln(r.repetiiiciones)))::numeric as r_total
    FROM public.activities a
    JOIN public.adaptive_rules_catalog r ON r.id = ANY(a.adaptive_rule_ids)
    CROSS JOIN public.clients c
    WHERE a.id = [ACTIVITY_ID] AND c.id = [CLIENT_UUID]
      AND (r.id = ANY(c.profile_rule_ids) OR r.id = ANY(c.injury_rule_ids))
),
dias_planificados AS (
    SELECT numero_semana, 'lunes' as dia, lunes as ej FROM public.planificacion_ejercicios
    WHERE actividad_id = [ACTIVITY_ID] AND lunes IS NOT NULL AND lunes::text != '{}'
    UNION ALL SELECT numero_semana, 'martes',    martes    FROM public.planificacion_ejercicios WHERE actividad_id = [ACTIVITY_ID] AND martes IS NOT NULL AND martes::text != '{}'
    /* ... demás días ... */
),
ejercicios_del_dia AS (
    SELECT DISTINCT d.numero_semana, d.dia, (ej_item->>'id')::int as ejercicio_id
    FROM dias_planificados d,
         jsonb_array_elements(CASE WHEN jsonb_typeof(d.ej->'ejercicios')='array' THEN d.ej->'ejercicios' ELSE '[]'::jsonb END) as ej_item
),
ejercicios_con_detalle AS (
    SELECT e.numero_semana, e.dia, ed.nombre_ejercicio,
           TRIM(BOTH '"' FROM ed.detalle_series::text) as ds_texto
    FROM ejercicios_del_dia e JOIN public.ejercicios_detalles ed ON ed.id = e.ejercicio_id
    WHERE ed.detalle_series IS NOT NULL
),
sets_unnested AS (
    SELECT numero_semana, dia, nombre_ejercicio, ds_texto,
           ordinality as set_num,
           TRIM(BOTH ' ()' FROM set_text) as set_clean
    FROM ejercicios_con_detalle,
         unnest(string_to_array(ds_texto, ';')) WITH ORDINALITY AS t(set_text, ordinality)
),
sets_adaptados AS (
    SELECT numero_semana, dia, nombre_ejercicio, ds_texto, set_num,
        '(' ||
            CASE WHEN split_part(set_clean,'-',1)::numeric=0 THEN '0'
                 ELSE (ROUND(split_part(set_clean,'-',1)::numeric * m.k_total / 2.5)*2.5)::text END
        || '-' || ROUND(split_part(set_clean,'-',2)::numeric * m.r_total)::text
        || '-' || ROUND(split_part(set_clean,'-',3)::numeric * m.s_total)::text
        || ')' as set_adaptado
    FROM sets_unnested, multiplicadores m
    WHERE set_clean != '' AND set_clean ~ '^[0-9]'
)
SELECT
    'S' || numero_semana || ' - ' || initcap(dia)        as "📅 Planificación",
    nombre_ejercicio                                       as "🏋️ Ejercicio",
    ds_texto                                               as "📋 Progresión Base",
    string_agg(set_adaptado, '; ' ORDER BY set_num)       as "⚡ Progresión Adaptada"
FROM sets_adaptados
GROUP BY numero_semana, dia, nombre_ejercicio, ds_texto
ORDER BY numero_semana,
    CASE dia WHEN 'lunes' THEN 1 WHEN 'martes' THEN 2 WHEN 'miercoles' THEN 3
             WHEN 'jueves' THEN 4 WHEN 'viernes' THEN 5 WHEN 'sabado' THEN 6 ELSE 7 END,
    nombre_ejercicio;

Formato de detalle_series: (peso_kg-reps-sets); (peso_kg-reps-sets)
Redondeo de peso: múltiplos de 2.5kg para compatibilidad con discos de gimnasio real.
